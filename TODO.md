## 当前最高优先级

# 照片寻句（用户拍照 / 上传照片后，自动寻句）

### Why
这是最符合小庄当前主线定位的下一个功能：

- 比纯文字输入更低门槛
- 更接近真实使用场景（看到一个画面就想配一句话）
- 更有传播性（照片 + 诗句天然适合分享）
- 直接强化“寻句是主产品”这条路线

一句话：

> 用户不是先想到“我要描述一个场景”，而是先看到一个场景。

所以“照片寻句”是对寻句主线最自然、最值得优先做的增强。

---

## 产品定义

用户可以：
- 直接拍一张照片
- 或上传一张已有照片

小庄基于图片内容理解场景、氛围、季节、人物关系或情绪，输出：
- 一句最贴切的古典诗文 / 古文
- 出处
- 白话解读
- 为什么适合这张照片

---

## Phase 1：先做 MVP

### 目标
让用户可以完成：

**上传图片 → 得到一句诗文结果**

### MVP 范围
- [ ] 寻句页支持上传图片
- [ ] 移动端支持拍照 / 从相册选择
- [ ] 后端接收图片并调用支持图片理解的模型
- [ ] 模型先描述图片，再生成寻句结果
- [ ] 复用现有寻句结果卡结构（quote/source/interpretation/resonance）
- [ ] 上传期间显示明确 loading 状态
- [ ] 图片过大时给出友好错误提示
- [ ] 模型失败 / 超时 / 图片无效时给出明确错误提示

### MVP 不做
- [ ] 不做图片裁剪
- [ ] 不做多图上传
- [ ] 不做历史记录
- [ ] 不做分享图生成
- [ ] 不做 OCR / 文字识别增强
- [ ] 不做图片风格滤镜或排版编辑

原则：

> 先验证“看图寻句”这件事是不是足够惊艳、足够自然。

## 技术实现方案

## 方案选择
采用：

> **前端上传单张图片 → API route 转发到多模态模型 → 统一返回现有寻句 JSON 结构**

原因：
- 改动最小
- 与现有寻句链路最一致
- 最容易先上线验证

## 错误与边界情况

- [ ] 文件不是图片

- [ ] 图片损坏 / 无法解析
- [ ] 图片内容过于模糊，模型难以判断
- [ ] 图片无明确主体
- [ ] 上游 vision 模型超时
- [ ] provider overloaded
- [ ] 用户重复点击提交
- [ ] 用户上传图片后又切换文本输入

对应要求：
- [ ] 所有失败都给出人话提示
- [ ] 不展示原始 provider 错误结构
- [ ] 不出现半截 JSON
